`use-clipboard.ts` wires up copy, cut, duplicate, and select-all as commands in the [command registry](./shortcuts-and-commands.mdx), plus a self-contained paste path that owns the browser's native `paste` event end to end.

## Copy, cut, duplicate, select all

| Shortcut | Action | Effect |
|---|---|---|
| `‚åò/Ctrl+C` | Copy | Snapshots the current selection to the internal clipboard and best-effort writes it to the OS clipboard. |
| `‚åò/Ctrl+X` | Cut | Copy, then delete the selection ‚Äî as **one** undo entry. |
| `‚åò/Ctrl+D` | Duplicate | Clones the selection with fresh ids and a small offset, selects the clones ‚Äî one undo entry. |
| `‚åò/Ctrl+A` | Select All | Selects every non-pattern layer. |

Copy, Cut, and Duplicate act on the **current selection** as-is ‚Äî a selected `pattern` layer (its movable `x`/`y`/`size` square, see [Tools ‚Üí Select ‚Üí Pattern squares](./tools.mdx#built-in-tools-select-v-pattern-squares)) is copyable/cuttable/duplicable exactly like any other layer type, so all three are no-ops only when *nothing* is currently selected. **Select All is the one deliberate exception**: it still excludes pattern layers on purpose ‚Äî a background-ish cover square joining every `‚åò/Ctrl+A` would make "select everything and move it" drag the background along too, so a pattern only joins a selection via a direct click (the [two-tier hit-test](./tools.mdx#built-in-tools-select-v-pattern-squares)) or the layer list. Select All doesn't read the current selection at all ‚Äî it always selects every non-pattern layer in the document, so it's a no-op only when the document has no non-pattern layers.

Every one of these chords is registered with `shift: false` ‚Äî `‚åò‚áßC` never copies, matching the pre-refactor behavior exactly.

## Where clones land

Both paste and duplicate share one clone technique (`cloneLayersWithFreshIds` from `@zpd/core`): fresh layer ids, and a **2mm cascade offset** applied to the clone's position ‚Äî so a repeated paste or duplicate never stacks exactly on top of its source. The clones are appended to the top of the layer stack and become the new selection.

## The OS clipboard envelope

Copy and cut write a versioned JSON envelope to the real OS clipboard via `navigator.clipboard.writeText`:

```ts
interface ZpdClipboardEnvelope {
  app: 'zpd';
  kind: 'layers';
  version: 1;
  layers: Layer[];
}
```

This is what makes copy/paste work **across zpd tabs** ‚Äî copy in one tab, switch to another, paste. Writing the envelope is best-effort: `navigator.clipboard.writeText` can reject, or the API can be entirely absent (e.g. an insecure context), and either case degrades silently to internal-clipboard-only ‚Äî copy/cut/duplicate/select-all never fail outright over a clipboard permission issue.

On paste, the OS clipboard's `text/plain` payload is parsed and checked against the envelope shape (`app`/`kind`/`version` markers, then the `layers` array is re-validated through core's defensive `parsePanelConfig` ‚Äî pattern layers included, since they're copyable per the exception above). Anything that isn't a recognized zpd envelope ‚Äî a URL, a sentence, JSON from an unrelated app, or a future/foreign envelope version ‚Äî is left **completely untouched**; a normal text paste into, say, the layer-rename field is never hijacked.

## Paste

`‚åò/Ctrl+V` is deliberately **not** a keydown-handled shortcut. `use-clipboard.ts` exposes no `handlePaste` at all, so there is nothing a keyboard handler could wire a `V` case to ‚Äî the editor's command registry lists an `edit-paste` entry purely as `displayOnly` (so the shortcuts overlay can still show "‚åòV ‚Äî Paste"), and its `run()` is never invoked. The real gesture rides the browser's native `paste` event, which `use-clipboard.ts` owns via a single `window` listener, checked in priority order:

1. **An image file on the OS clipboard** ‚Äî e.g. copied from an image viewer or another app. Imported as a new `image` layer via the same `importImageFile` pipeline the Add image‚Ä¶ action and drag-drop import use.
2. **Text on the OS clipboard** ‚Äî parsed as a zpd envelope. A valid envelope pastes those layers (fresh ids, cascade offset, selected). Non-envelope text is left alone entirely, **including skipping the internal-clipboard fallback below** ‚Äî the OS clipboard's current content always wins over a stale in-app copy. The one exception: if *this session's own* `writeText` from the most recent copy/cut hasn't resolved yet (a fast copy-then-paste can race the async write), the read text may simply predate that write ‚Äî in that case paste falls through to the internal snapshot instead of silently doing nothing.
3. **The internal same-session clipboard** ‚Äî the last resort, used when the OS clipboard has neither an image nor any text at all (e.g. `writeText` was denied on copy).

Paste is ignored while an `<input>`, `<textarea>`, `<select>`, or a `contenteditable` element is focused, so a normal text paste into an editable field is never intercepted.
